# EatSnap 需求文档

> 版本：v0.1（初版需求）  日期：2026-07-08

## 1. 项目简介

EatSnap 是一个个人使用的「拍食物照片 → 多模态大模型识别 → 计算并记录每日热量」的小工具。目标是用**最少操作**完成每日三餐热量记账，取代手动查阅食物成分表、心算热量的繁琐流程。

## 2. 背景与价值

- **痛点**：手动记录饮食热量需要逐项查表、估算克重、心算求和，坚持成本高；现有 App 往往重交互、需注册、多端同步复杂。
- **价值**：拍一张照片即可获得结构化的「食物名 + 重量(g) + 热量(kcal)」并自动归档到当日当餐，配合本机静态看板回看趋势。
- **复用基础**：本项目前期已验证可通过多模态模型对食物图完成「识别拆分 → 重量预估 → 热量计算 → 合计」的完整链路（沿用同一套提示词约束）。

## 3. 用户角色与场景

- **角色**：单一用户（开发者本人）。单人使用，无多用户、无租户概念。
- **典型场景**：
  - 用餐前/后用手机或电脑浏览器打开采集页，选餐别、拍食物照片、上传。
  - 一段时间后在服务端触发识别，生成当日热量记录与静态报告。
  - 随时本地双击打开 HTML 报告查看今日及近期热量趋势。

## 4. 功能需求清单

| 编号 | 名称 | 描述 |
|------|------|------|
| FR1 | 网页采集拍照 | 浏览器调本机/手机摄像头拍照；选择餐别（早 / 午 / 晚 / 加餐）归类；支持填写备注（如手填食物名做补录提示）。 |
| FR2 | 图片上传与同步 | 拍完 POST 上传到服务端；服务端将图片落盘到本机 iCloud 目录，借助 iCloud 自动云端备份，无需额外 OAuth 开发。 |
| FR3 | 识别与热量计算 | 手动触发批量识别：扫描监控目录中未处理的图片，逐张调用多模态大模型，输出结构化「食物名、重量(g)、热量(kcal)、合计」。 |
| FR4 | 记录存储 | 识别结果写入本机 JSON，按日（`YYYY-MM-DD.json`）归档，记录每餐的条目、餐别、图片路径、识别时间、所用模型、总热量。 |
| FR5 | 静态报告 | 生成本机静态 HTML 报告，展示每日三餐条目与热量明细、日累计与近期趋势。纯静态，断网可看。 |
| FR6 | 模型可配置 | 通过配置文件切换默认识别模型及各厂商 API Key；初版仅支持「切换默认模型」，不做并行比对或微调。 |

## 5. 非功能需求

| 维度 | 要求 |
|------|------|
| 隐私 | 食物图片与热量记录仅存本机 + 个人 iCloud，不上传到第三方托管；识别调用外部 API 时图片需经多方大模型服务过一遍（用户已知悉并接受）。 |
| 维护性 | 个人工具，低维护：依赖少、部署简单（本机跑 FastAPI）；配置项清晰。 |
| 离线查看 | 静态报告本地双击即可打开，无需联网。 |
| 跨生态 | 采集端为网页，任何带摄像头的设备浏览器均可拍照上传，不绑死某家网盘或某品牌设备。 |
| 性能 | 个人量级（每日 ≤ 几张图），对吞吐无要求；单张识别延迟以多模态 API 实际响应为准。 |

## 6. 约束与假设

- **依赖外部多模态 API**：识别能力依赖第三方大模型服务，需联网与有效 API Key。
- **Apple 生态假设**：图片落盘到 iCloud 目录的前提是本机已登录同一 Apple ID 且 iCloud Drive 已启用；该路径在不同环境下可能略有差异，需做成配置项。
- **触发方式**：初版为手动触发，不自动定时；后续可加定时轮询作为增强。
- **单用户**：不做认证鉴权、多租户、数据隔离等企业级需求。

## 7. 范围边界

**纳入范围**：网页采集、上传存盘、手动触发识别、JSON 记录、静态 HTML 报告、模型配置切换。

**初版不做**：自动定时识别、多模型并行比对、数据集积累与微调、移动端原生 App、登录鉴权、云端数据库、多用户协作、营养素细分（仅热量为主）。
