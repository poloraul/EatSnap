# EatSnap 产品设计文档

> 版本：v0.1  日期：2026-07-08  配套：《需求文档》《实现路径规划》

## 1. 整体架构

数据流（文字版架构图）：

```
[tools/shot.html（手机 Safari, 独立 HTML）]
        │  navigator.share({files: [File]})  → "存储到文件"
        ▼
[iCloud Drive/EatSnap/]   ←── iCloud 跨设备同步 ───→
        │
        ▼  （Mac 本机 iCloud 目录）
[Mac: ~/Library/Mobile Documents/com~apple~CloudDocs/EatSnap/images/]
        │  根目录扁平文件（未归位）
        ▼  ingest_pending() 自动按文件名归位
[images/{date}/{meal}/]
        │  手动触发：python -m server identify [date]
        ▼
[多模态识别引擎]  ──HTTP──▶ 多模态大模型 API(默认 Claude，可配置切换)
        │  结构化 JSON(食物名/重量/热量/合计)
        ▼
[JSON 记录库(records/)]
        │
        ▼
[静态 HTML 报告]  ──▶ reports/index.html（本地）+ 推 master/docs/（GitHub Pages）
```

> 关键解耦：采集端 `tools/shot.html` 与服务端**完全独立**——零网络依赖、`file://` 即可打开、
> 跨生态（iOS/Android/桌面浏览器通用）。iOS Safari 拍照后通过系统分享面板写入 iCloud Drive，
> Mac 端识别引擎只负责归位 + 识别 + 记录 + 报告。

## 2. 关键目录约定

## 2. 关键目录约定

| 用途 | 路径 | 说明 |
|------|------|------|
| 图片根目录 | `~/Library/Mobile Documents/com~apple~CloudDocs/EatSnap/images/` | iCloud 同步，Mac 与 iPhone 共享 |
| 归位后子目录 | 上面的 `.../images/{YYYY-MM-DD}/{早|午|晚|加餐}/` | server 识别的源 |
| 扁平文件（未归位） | 上面的 `.../images/{YYYY-MM-DD}_{meal}_{note}_{HHMMSS}.jpg` | iOS 端拍完直接落盘，由 `ingest_pending()` 自动归位 |
| JSON 记录库根目录 | `records/`（项目内，运行期产物，gitignore） | 按日一个文件 |
| 单日记录 | `records/{YYYY-MM-DD}.json` | 当日三餐所有条目 |
| 静态报告目录 | `docs/reports/`（项目内，已通过 GitHub Pages 发布） | 纯 HTML |
| 报告入口 | `docs/index.html`（项目根的 docs/ 下） | Pages 根 URL 直接打开 |
| 单日报告页 | `docs/reports/{YYYY-MM-DD}.html` | 当日三餐明细 |
| 报告内嵌图 | `docs/reports/images/{...}` | 复制自 iCloud 目录，离线可看 |
| 模型/服务配置 | `config.yaml` 或 `.env` | 见第 7 节 |

> 可选：在 `~/Library/Mobile Documents/.../EatSnap/` 下用一个 `eating` 标记文件或 records 自身判断哪些图片「已处理」，避免重复识别。

## 3. 采集端设计

### 3.1 主路径：手机独立 HTML（`tools/shot.html`）

**核心思路：手机端零服务端依赖。**

极简单页（与 `web/index.html` 同款 UI），无登录、无后端状态：
1. 选择餐别：单选——早 / 午 / 晚 / 加餐。
2. 调用摄像头预览（`getUserMedia`），取后置摄像头优先（手机）。
3. 拍照按钮 → 抓取一帧 → 本地预览缩略图。
4. 备注输入框（可选），随图编码进文件名（6 位 hex hash）。
5. 「保存到 iCloud」按钮 → `navigator.share({files: [File]})` → 系统分享面板 → 选「存储到文件」→ 选 `iCloud Drive/EatSnap/` 根目录。
6. 拍照成功提示，可继续拍下一张。

> **目录自动归位**：iOS Safari 写文件时**无法预设子目录**，但拍出的文件名已编码
> `{date}_{meal}_{note}_{time}.jpg`，Mac 端 `server/storage.py:ingest_pending()` 扫描根目录扁平文件
> 按文件名解析、自动 `mv` 到 `{date}/{meal}/` 子目录。

### 3.2 Fallback：浏览器上传到 FastAPI（`web/index.html`）

保留作为兜底（如 Android Chrome / 桌面浏览器无 `navigator.canShare` 支持时）：

1. 选择餐别 → 调摄像头 → 拍照 → 备注
2. `POST /upload`：multipart，字段 `meal`、`remark`、`file`
3. 服务端 `save_upload()` 落盘到 `{date}/{meal}/` 子目录

> 注意：手机端走 FastAPI 需要 `localhost` 或 HTTPS 域名（浏览器 `getUserMedia` 限制）。
> 局域网 IP 在多数浏览器会被拒绝授权摄像头。

## 4. 识别流程设计

### 4.1 触发方式
- 初版：手动触发。两种入口任选其一：
  - HTTP：`POST /identify?date=YYYY-MM-DD`（或全部）。
  - CLI：`python -m server identify --date YYYY-MM-DD`。
- **每次 identify 触发前自动调用 `ingest_pending()`**：扫描根目录扁平文件，按文件名解析日期/餐别，自动 `mv` 到 `{date}/{meal}/` 子目录。
- 扫描指定日的图片目录，跳过已处理图片（依据 records 中已记录的图片路径或独立状态文件）。

### 4.2 单张识别
- 读取图片 → 调可配置多模态模型 → 强制返回结构化结果。
- 沿用本项目已验证的提示词约束（识别与拆分 / 重量预估 g / 热量计算 kcal / 合计），并将输出约束为 JSON：
  ```json
  {
    "items": [
      {"name": "面包", "weight_g": 50, "calories_kcal": 300},
      {"name": "鸡蛋", "weight_g": 70, "calories_kcal": 75}
    ],
    "total_kcal": 375,
    "notes": ""
  }
  ```
- 重量不确定的条目在结果里以数值+「约」标识（前端/报告展示时体现）。

### 4.3 容错
- API 失败/JSON 解析失败：重试 N 次，仍失败则标记该图为「待人工复核」，记录错误日志，不阻断其它图。
- 识别完成后写回 records JSON（追加到当日当餐的条目数组），并记录所用模型名与识别时间。

## 5. 记录数据结构（JSON Schema 草案）

`records/{YYYY-MM-DD}.json`：

```json
{
  "date": "2026-07-08",
  "meals": {
    "早": [],
    "午": [
      {
        "image": "images/2026-07-08/午/20260708-1230_ab12.jpg",
        "remark": "",
        "identified_at": "2026-07-08T13:00:00",
        "model": "claude",
        "items": [
          {"name": "炸酱面", "weight_g": 250, "calories_kcal": 400},
          {"name": "黄瓜丝", "weight_g": 30, "calories_kcal": 5}
        ],
        "total_kcal": 405,
        "status": "ok"
      }
    ],
    "晚": [],
    "加餐": []
  },
  "daily_total_kcal": 405
}
```

字段说明：
- `meals` 四个固定 key；`加餐` 可多次留多张图。
- `status`：`ok` / `review`（待人工复核）/ `error`。
- `daily_total_kcal`：当日所有条目合计，随写入刷新。

## 6. 报告设计

- `reports/index.html`：首页看板
  - 顶部：今日三餐热量小计与日合计。
  - 中部：近 7/30 日每日总热量柱状/折线（纯静态，用内联 SVG 或轻量 CDN 图表，或直接表格）。
  - 底部：各日子页链接列表。
- `reports/{YYYY-MM-DD}.html`：单日明细
  - 按早/午/晚/加餐分组，展示每张图缩略图 + 条目表（名称/重量/热量）+ 小计。
  - 标注重量为「约」的条目。
- 纯静态：双击文件即可在浏览器打开，断网可看。生成由 Jinja2 模板渲染后落盘 HTML。

## 7. 多模型配置设计

配置文件（`config.yaml`，敏感 key 可走 `.env`）：

```yaml
default_model: claude
models:
  claude:
    provider: anthropic
    model: claude-sonnet-6   # 实际所用模型 ID
    api_key_env: ANTHROPIC_API_KEY
  openai:
    provider: openai
    model: gpt-4o
    api_key_env: OPENAI_API_KEY
# 服务端路径配置
paths:
  images_root: "~/Library/Mobile Documents/com~apple~CloudDocs/EatSnap/images"
  records_dir: "./records"
  reports_dir: "./reports"
```

- 初版「可配置」范围：仅**切换默认模型**（改 `default_model` 重启生效）；各模型走统一的多模态调用抽象（输入图+提示词，输出上述 JSON）。
- 不做：多模型并行比对、数据集积累与微调、自动选模型。

## 8. 服务端接口草案

| 方法 | 路径 | 作用 |
|------|------|------|
| POST | `/upload` | 接收图片(餐别/备注)，落盘 iCloud 目录 |
| POST | `/identify?date=` | 手动触发指定日批量识别 |
| GET  | `/records/{date}` | 读取当日 JSON 记录 |
| POST | `/report?date=` | 触发生成静态报告（或 CLI） |

> 简单工具可只暴露 `/upload` + `/identify` + CLI 生成报告，其余后续补。
