# EatSnap 产品设计文档

> 版本：v0.1  日期：2026-07-08  配套：《需求文档》《实现路径规划》

## 1. 整体架构

数据流（文字版架构图）：

```
[采集网页(浏览器摄像头)]
        │  HTTP POST(multipart, 带餐别/备注)
        ▼
[FastAPI 服务端]
        │  落盘图片
        ▼
[本机 iCloud 目录]  ──iCloud 自动云端备份──▶ (云端)
        │
        │  手动触发(/identify 或 CLI)
        ▼
[多模态识别引擎]  ──HTTP──▶ 多模态大模型 API(默认 Claude，可配置切换)
        │  结构化 JSON(食物名/重量/热量/合计)
        ▼
[JSON 记录库(records/)]
        │
        ▼
[静态 HTML 报告生成] ──▶ reports/index.html(本地双击可看)
```

> 跨生态说明：采集端是网页，任何带摄像头设备的浏览器都能拍并 POST 到服务端地址；图片最终统一落盘到本机 iCloud 目录做备份。不绑死某家网盘 OAuth。

## 2. 关键目录约定

| 用途 | 路径 | 说明 |
|------|------|------|
| 图片监控（落盘）根目录 | `~/Library/Mobile Documents/com~apple~CloudDocs/EatSnap/images/` | 可配置；iCloud 自动同步 |
| 单张图片 | 上面的 `.../images/{YYYY-MM-DD}/{早|午|晚|加餐}/{时间戳或uuid}.jpg` | 按日+餐别分层 |
| JSON 记录库根目录 | `records/`（项目内，运行期产物，gitignore） | 按日一个文件 |
| 单日记录 | `records/{YYYY-MM-DD}.json` | 当日三餐所有条目 |
| 静态报告目录 | `reports/`（项目内，运行期产物，gitignore） | 纯 HTML |
| 报告入口 | `reports/index.html` | 近期趋势 + 各日链接 |
| 单日报告页 | `reports/{YYYY-MM-DD}.html` | 当日三餐明细 |
| 模型/服务配置 | `config.yaml` 或 `.env` | 见第 7 节 |

> 可选：在 `~/Library/Mobile Documents/.../EatSnap/` 下用一个 `eating` 标记文件或 records 自身判断哪些图片「已处理」，避免重复识别。

## 3. 网页采集端设计

极简单页，无登录、无后端状态，纯前端 + 一个上传接口：

1. 选择餐别：单选——早 / 午 / 晚 / 加餐。
2. 调用摄像头预览（`getUserMedia`），取后置摄像头优先（手机）。
3. 拍照按钮 → 抓取一帧 → 本地预览缩略图。
4. 备注输入框（可选）：手写食物名/份量提示，随图上传，供识别参考。
5. 上传按钮 → `multipart` POST 到 `/upload`，字段：餐别、备注、图片二进制。
6. 上传成功提示，可继续拍下一张。

> 手机上：用浏览器（Safari/Chrome）打开页面即可，同样走摄像头授权 + 上传，无需 App。

## 4. 识别流程设计

### 4.1 触发方式
- 初版：手动触发。两种入口任选其一：
  - HTTP：`POST /identify?date=YYYY-MM-DD`（或全部）。
  - CLI：`python -m server.identify --date YYYY-MM-DD`。
- 扫描指定日的图片目录，跳过已处理图片（依据 records 中已记录的图片路径或独立状态文件）。

### 4.2 单张识别
- 读取图片 → 调可配置多模态模型 → 强制返回结构化结果。
- 沿用本项目已验证的提示词约束（识别与拆分 / 重量预估 g / 热量计算 kcal / 合计），并将输出约束为 JSON：
  ```json
  {
    "items": [
      {"name": "面包", "weight_g": 50, "calories_kcal": 300},
      {"name": "鸡蛋", "weight_g": 70, "calories_kcal": 75}
    ],
    "total_kcal": 375,
    "notes": ""
  }
  ```
- 重量不确定的条目在结果里以数值+「约」标识（前端/报告展示时体现）。

### 4.3 容错
- API 失败/JSON 解析失败：重试 N 次，仍失败则标记该图为「待人工复核」，记录错误日志，不阻断其它图。
- 识别完成后写回 records JSON（追加到当日当餐的条目数组），并记录所用模型名与识别时间。

## 5. 记录数据结构（JSON Schema 草案）

`records/{YYYY-MM-DD}.json`：

```json
{
  "date": "2026-07-08",
  "meals": {
    "早": [],
    "午": [
      {
        "image": "images/2026-07-08/午/20260708-1230_ab12.jpg",
        "remark": "",
        "identified_at": "2026-07-08T13:00:00",
        "model": "claude",
        "items": [
          {"name": "炸酱面", "weight_g": 250, "calories_kcal": 400},
          {"name": "黄瓜丝", "weight_g": 30, "calories_kcal": 5}
        ],
        "total_kcal": 405,
        "status": "ok"
      }
    ],
    "晚": [],
    "加餐": []
  },
  "daily_total_kcal": 405
}
```

字段说明：
- `meals` 四个固定 key；`加餐` 可多次留多张图。
- `status`：`ok` / `review`（待人工复核）/ `error`。
- `daily_total_kcal`：当日所有条目合计，随写入刷新。

## 6. 报告设计

- `reports/index.html`：首页看板
  - 顶部：今日三餐热量小计与日合计。
  - 中部：近 7/30 日每日总热量柱状/折线（纯静态，用内联 SVG 或轻量 CDN 图表，或直接表格）。
  - 底部：各日子页链接列表。
- `reports/{YYYY-MM-DD}.html`：单日明细
  - 按早/午/晚/加餐分组，展示每张图缩略图 + 条目表（名称/重量/热量）+ 小计。
  - 标注重量为「约」的条目。
- 纯静态：双击文件即可在浏览器打开，断网可看。生成由 Jinja2 模板渲染后落盘 HTML。

## 7. 多模型配置设计

配置文件（`config.yaml`，敏感 key 可走 `.env`）：

```yaml
default_model: claude
models:
  claude:
    provider: anthropic
    model: claude-sonnet-6   # 实际所用模型 ID
    api_key_env: ANTHROPIC_API_KEY
  openai:
    provider: openai
    model: gpt-4o
    api_key_env: OPENAI_API_KEY
# 服务端路径配置
paths:
  images_root: "~/Library/Mobile Documents/com~apple~CloudDocs/EatSnap/images"
  records_dir: "./records"
  reports_dir: "./reports"
```

- 初版「可配置」范围：仅**切换默认模型**（改 `default_model` 重启生效）；各模型走统一的多模态调用抽象（输入图+提示词，输出上述 JSON）。
- 不做：多模型并行比对、数据集积累与微调、自动选模型。

## 8. 服务端接口草案

| 方法 | 路径 | 作用 |
|------|------|------|
| POST | `/upload` | 接收图片(餐别/备注)，落盘 iCloud 目录 |
| POST | `/identify?date=` | 手动触发指定日批量识别 |
| GET  | `/records/{date}` | 读取当日 JSON 记录 |
| POST | `/report?date=` | 触发生成静态报告（或 CLI） |

> 简单工具可只暴露 `/upload` + `/identify` + CLI 生成报告，其余后续补。
