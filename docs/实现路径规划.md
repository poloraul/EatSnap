# EatSnap 实现路径规划

> 版本：v0.1  日期：2026-07-08  配套：《需求文档》《产品设计文档》
> 本文档给出从零到 MVP 的阶段化落地顺序，**不写具体代码**，作为后续编码的执行清单。

## 1. 目标

按阶段把 EatSnap 从空仓库落地到可用 MVP：网页拍照 → iCloud 落盘（备份）→ 手动触发多模态识别 → JSON 记录 → 静态报告。每阶段交付可独立验证的最小闭环。

## 2. 推荐目录结构（项目根）

```
EatSnap/
├── docs/                  # 文档（本次产出）
├── server/                # FastAPI 服务端
│   ├── main.py            # 入口 + /upload, /identify 等接口
│   ├── config.py          # 加载 config.yaml / .env
│   ├── storage.py         # iCloud 落盘 + 图片扫描
│   ├── identify.py        # 多模态识别抽象 + 手动触发
│   ├── records.py         # JSON 记录读写
│   └── report.py          # 静态 HTML 报告生成
├── web/                   # 采集单页(index.html, 纯前端)
├── templates/             # 报告 Jinja2 模板
├── config.yaml            # 模型与路径配置(示例, key 走 .env)
├── .env.example           # API Key 模板
├── requirements.txt
├── .gitignore
└── (运行期产物, gitignore)
    ├── records/           # 每日 JSON
    └── reports/           # 静态 HTML
```

> 图片落盘在 iCloud 目录（`~/Library/.../EatSnap/images/`），不在项目内。

## 3. 阶段划分

### 阶段 0：仓库骨架与文档就位（本次完成）
- 建立 `docs/` 三份文档。
- 初始化 `.gitignore`（忽略 `records/`、`reports/`、`.env`、`__pycache__/`）。
- 产出物：本文档三件套 + gitignore 占位。

### 阶段 1：服务端最小闭环
- 目标：能接收图片并落盘到 iCloud 目录，记录结构就绪。
- 任务：
  - `server/config.py`：加载配置（图片根目录、records/reports 目录、默认模型）。
  - `server/storage.py`：按 `images/{date}/{餐别}/` 落盘上传图片；扫描目录列出图片。
  - `server/main.py`：`POST /upload`（餐别/备注/图片），写入磁盘并返回保存路径。
  - `server/records.py`：当日 JSON 的读/追加/写接口；空骨架先就位。
- 验证：用 `curl` 或网页 POST 一张图，确认 iCloud 目录出现文件，`records/{date}.json` 被初始化。

### 阶段 2：网页采集页
- 目标：浏览器能调摄像头拍照并上传。
- 任务：
  - `web/index.html`：餐别单选 + 摄像头预览（`getUserMedia`，手机优先后置）+ 拍照 + 缩略图预览 + 备注框 + 上传按钮。
  - 上传走阶段 1 的 `POST /upload`。
  - FastAPI 把 `web/` 作为静态目录托管，方便手机浏览器打开本机地址。
- 验证：手机浏览器打开 `http://<本机IP>:<port>/`，能拍照并上传，服务器落盘成功。

### 阶段 3：识别能力
- 目标：手动触发，把未处理图片识别成结构化结果写回 JSON。
- 任务：
  - `server/config.py` 补全多模型配置（`config.yaml` + `.env` 中的 key）。
  - `server/identify.py`：多模态调用抽象——输入图片路径+备注+提示词，返回第 5 节 JSON；按 `default_model` 选实现（初版接 Claude，附 OpenAI 占位）。
  - 提示词沿用本项目已验证约束（识别拆分/重量 g/热量 kcal/合计 + 强制 JSON 输出）。
  - `POST /identify?date=` 与 CLI（`python -m server.identify`）：扫描该日图片，跳过已处理（查 records 已记的图片路径），逐张识别、重试、写回记录、维护 `status` 与 `model`。
- 验证：上传一两张食物图，触发 `/identify`，`records/{date}.json` 出现条目、`total_kcal`、`model`、`identified_at`。

### 阶段 4：静态报告生成
- 目标：生成本机可双击打开的 HTML 看板。
- 任务：
  - `templates/index.html`、`templates/day.html`：Jinja2 模板。
  - `server/report.py`：读 records 渲染首页（近 N 日趋势、今日合计）与各日明细页（三餐分组 + 缩略图 + 条目表）。
  - CLI 或 `POST /report?date=` 触发；产物写入 `reports/`。
- 验证：`reports/index.html` 本地双击打开，能看到当日明细与近期走势；图片缩略图用相对路径或 base64 内联保证离线可看。

### 阶段 5（可选打磨）
- 已处理标记：独立状态文件或在 records 内沉淀图片处理状态，避免重复识别。
- 补录/编辑：在 records JSON 上做人工纠错接口或网页编辑页（重量「约」的条目重点保留可改）。
- 趋势图：首页趋势换成轻量内联可视化（SVG/Canvas），纯静态。
- 自动轮询：可选加定时任务替代手动触发。

## 4. 关键技术依赖候选

| 依赖 | 用途 |
|------|------|
| FastAPI + Uvicorn | 服务端与接口 |
| httpx | 调用多模态 API |
| Pillow | 图片基础处理/缩略图 |
| Jinja2 | 静态报告模板渲染 |
| python-dotenv / PyYAML | 配置与 API Key |
| (前端原生) | `getUserMedia` 摄像头，无需框架 |

## 5. 风险与取舍

| 风险/取舍 | 说明 | 应对 |
|-----------|------|------|
| 重量估算波动 | 多模态对相同食物在不同光线/角度下克重估算不稳定 | 记录中保留 `model` + 重量「约」标注，预留人工纠错入口 |
| iCloud 路径差异 | 不同环境/账号下 iCloud 路径可能不同 | 路径做成配置项，提供默认值 |
| 手动触发 vs 自动 | 自动定时更省心但增加复杂度 | 初版手动，后续按需加定时（阶段 5） |
| 跨端外网访问 | 手机上传需本机服务在局域网可达 | FastAPI 监听 `0.0.0.0`，使用本机局域网 IP；外网非目标 |
| API 成本/限速 | 多模态调用按量计费 | 个人量级、手动触发，单日少量调用；必要时缓存/复用结果 |
